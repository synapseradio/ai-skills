Before walking the futures, three questions are worth answering in the open, because the answers change how much machinery this needs.

What bounds the decision: the reframe is sequenced first, and three other in-flight changes explicitly declare a dependency on the typed substrate it provides — `scope-context-by-actor-subtree` and `validate-program-at-load` name it in their own sequencing sections, and `define-runtime-supervision-model` leans on the typed single-writer state to argue serialized dispatch is race-free (`openspec/changes/scope-context-by-actor-subtree/proposal.md`, `openspec/changes/validate-program-at-load/proposal.md`, `openspec/changes/define-runtime-supervision-model/proposal.md:37`). Two other open changes, `clarify-boundary-enforcement` and `clarify-module-resolution`, don't touch it at all — they're a separate track. Nothing has actually shipped against either model yet: `packages/runtime/contracts/src/types.ts` still carries the full string-addressed `World`/`Namespace`/`ContextPath` shape and the old `InferenceRef.fn: string`, and `define-program-ir` — the JSON-IR version of the parser-output premise — already lost this argument once, dissolved into `validate-program-at-load` in an earlier commit. So this isn't a live migration to unwind; it's a decision made at the cheapest possible moment, before any of the three dependents rewrite a line.

What I could verify rather than assume: nothing in the current runtime actually depends on actor definitions being JSON-serializable today. A search across `packages/runtime` for the calls that would need it — snapshot persistence, `structuredClone`, `JSON.stringify`/`parse` on a definition — turns up nothing outside generated `dist/` and tests. The README states plainly that the first authoring surface is "a TypeScript embedded DSL," that the program the runtime executes today is hand-authored, not parsed, and that the `legacy/` corpus of real `.in`-syntax programs is explicitly reference material never executed by the runtime. So the parser-output model's central premise — a string tree that a future parser fills in — isn't backed by a parser that exists, a persistence path that exists, or a corpus that runs. It's backed by a comment.

That leaves one real unresolvable, not several. It isn't whether a `.in` parser ever gets built — the proposal already resolves that: if one is built, it becomes a producer of the typed model, not the model's definition, so its eventual existence doesn't argue for keeping the string-addressed shape as the foundation. It isn't the typed builder's ergonomics (DL5) either — that question exists in both worlds and doesn't flip the recommendation either way. The one axis that actually changes what you'd do is timing: does a real need to persist and resume a running actor tree show up before the typed substrate, the validator, and the scope work have matured — forcing the function-reference-in-definition problem to be solved under deadline pressure — or does it stay a distant, deliberately-deferred concern the way it is today?

Two futures follow from that one axis, and they're worth telling as one story rather than two. In the one where persistence stays distant, the reframe lands clean: `scope-context-by-actor-subtree` gets a typed substrate to specify visibility against instead of a dotted-path guess, `validate-program-at-load` inherits a genuinely narrower residue to check because the type system already carries unbound-reference and out-of-scope errors at author time, and the three-proposal rebase this quarter was already the right bet. In the harsher one, a real session-resume requirement lands early — mid-migration, before the substrate settles — and someone has to retrofit a way to serialize an actor definition that now holds a live function reference, under the exact time pressure the design doc names as the one-way door it's knowingly walking through. That's a real cost, but it isn't a fatal one: the separation the reframe already commits to — state stays data, definitions stay code and get re-supplied at resume — is the standard shape of that fix, just designed in a hurry instead of on schedule.

What holds in both tellings matters more than which one arrives. The string-addressed model isn't a neutral fallback sitting there for free — it already lost once inside this same repo when `define-program-ir` was dissolved, nothing today reads or writes an actor definition as JSON, and the one corpus that would validate the parser-output premise is explicitly retired reference material. So "keep the parser-output model as the foundation" isn't actually the safe, low-commitment choice in either future; it's the choice that keeps paying a cost — no type-carried no-meld, a load-time validator stuck re-checking what the compiler could catch, three dependent proposals without the substrate they're already written against — for a benefit (deferred serializability) that nothing currently collects on.

The hinge to watch for is narrow and checkable: if a persistence, snapshot, or session-resume proposal gets scaffolded under `openspec/changes/` before `validate-program-at-load` and `scope-context-by-actor-subtree` land, that's the early signal the harsher future is arriving, and it's the moment to stop and design the function-reference serialization seam deliberately rather than let the three-proposal rebase discover it mid-flight. Its absence through the next couple of landings is itself the confirming signal for the easier future — not proof, but the best evidence available before the fact.

Commit to the reframe as the foundation now, and let the three dependents rebase onto it. It's the move that comes out ahead in the future you're likely in, and in the harsher one it costs a scramble you were already told about and already have a shape for, not a redesign. Where it runs weakest is exactly there — if persistence needs turn up before the substrate and the validator mature, budget for the retrofit deliberately rather than discovering it under a deadline. Short of that, this isn't a coin flip between two live foundations; the parser-output model already isn't carrying its own weight in this codebase today.
