Two of your three options don't actually compete on equal footing once checked against your own architecture docs, so the walk resolves further than the pilot left it.

`thinking_primitives.in` declares four sequences: `explore_possibility`, `consider_alternatives`, `deepen_understanding`, `step_back_and_reflect`. The pilot only ported `explore_possibility`, and even that one carries a `when: depth greater_than 0` guard between its first and second step. `consider_alternatives` loops an `each:` over branches with a persona switch inside the loop body. `deepen_understanding` loops `each: [1, 2, 3] as |level| when level <= max_depth` between two `@curious_explorer` steps. Most of the module already contains runtime-visible branching and looping between persona-voiced steps. The pilot measured only the single straight-line case.

That fact kills the fused option outright, and principle 2 forces the kill: the runtime decides transitions on typed data, and a guard reads committed state. A single fused call produces one value only after the model has silently made every intermediate decision inside its own completion, so no guard has anything committed to read between steps. Collapse `consider_alternatives` or `deepen_understanding` into one call and the port loses the exact branching the source already has, trading it for one opaque completion. So "collapse into one fused call" drops out as a live option, already disqualified for three of the module's four sequences. The fourth, `explore_possibility`, escapes the disqualification only when depth lands at 0.

So the real choice narrows to always chain against support both, with fusion as an optimization. What decides between them turns out to hinge on how the runtime grows, not on what it needs today, and on two things you don't yet know the answer to.

One: whether the INDRA-to-BAML compiler, still planned rather than built (`docs/toolchain.md` calls it "design intent"), ends up strict, one declared inference point mapping to exactly one generated function always, or whether its optimizer earns permission to quietly collapse a run of steps that never branches into one generated call. Two: whether chained, multi-step persona sequences turn out cheap enough in production, once real concurrency and real cost show up, that nobody ever feels pressure to fuse anything. Four futures fall out of those two questions. None collides with anything already true, so all four survive triage.

Where the compiler stays strict and cost never bites, chaining just works. Every PRISM sequence ports as a straight run of typed calls, the AST keeps every persona-voiced step legible, and nothing about `thinking_primitives` ever needs a second form. This tile hands you the recommendation with no asterisk attached.

Cost pressure without compiler relief tells a rougher story. "Always chain, no exceptions" runs into real trouble here. Long sequences add up, three to five round trips per exploration, times however many actors run one concurrently, and if the protocol never gives an author a sanctioned way to collapse a run of steps proven safe, the pressure finds another outlet: someone writes a fused prompt by hand in host code to save money. That move recreates the exact leak principle 4 already names as the risk: "if composition drifts into hand-written host functions, INDRA stops being a language and becomes a library." A policy of never fusing, anywhere, at any layer, survives the axis but not the incentive it creates.

Compiler relief without cost pressure looks almost boring by comparison. Fusion shows up anyway, as a backend optimization nobody asked for out of necessity. The author still writes chained steps. The compiler just notices a run of steps that never branches and emits one BAML function instead of three. Nothing about the protocol or the port changes. Only the generated artifact gets cheaper.

Put both pressures in the same room and fusion stops being optional. It becomes the thing standing between the runtime and a real cost wall, so its correctness, whether it actually detects "no guard, no loop, no persona conflict" reliably, turns from something nice to have into something worth testing directly.

| | Compiler stays strict (1:1) | Compiler may fuse |
|---|---|---|
| Cost stays comfortable | Chaining just works, no asterisk | Fusion appears as an invisible, optional optimization |
| Cost bites | "Never fuse" gets exposed; pressure leaks into hand-written host code | Fusion turns load-bearing, not just a nicety |

What holds across all four: the AST and the protocol always keep every persona-voiced step as its own declared unit, with its own `as:`, its own typed return, and its own place for a guard to sit between it and the next step. That part of the answer doesn't wait on how the compiler or production load turn out. Principle 2 and D9 force it already, today, for a module with the shape `thinking_primitives` already has.

Two watchpoints flip the rest. Watch the compiler design doc once the INDRA-to-BAML pipeline actually gets built (`toolchain.md` still lists it as planned, not started). Commitment to strict 1:1 codegen puts you choosing between the top two tiles. Reservation of an optimizer pass puts you choosing between the bottom two. Separately, once real programs run against the live model at real concurrency, watch p95 turn latency and the token cost each turn incurs for actors partway through a sequence. Parity with a single call of equivalent output keeps you in the left column. A real gap pushes you into the right one.

Support both, but not the way "both" usually gets read. An author only ever writes chained steps, the same shape `thinking_primitives.in` already has: one typed call per persona-voiced step, with data passed between them through the runtime. Fusion never becomes a second thing an author reaches for. It stays an internal compiler move, applied only where the AST can prove no guard, no loop, and no persona conflict sits between the steps being merged, and it never touches a sequence shaped like `consider_alternatives` or `deepen_understanding` at all. That version of "both" survives every tile: it degenerates cleanly to plain chaining wherever fusion never gets built or never gets needed, and it absorbs cost pressure without anyone rolling a fused prompt by hand outside the language the day that pressure shows up. This walk exposes the real risk as the blanket ban on ever fusing anywhere, not fusion itself: the tile where cost bites and the compiler stays strict turns a principled stance into a reason to defect from the protocol.
