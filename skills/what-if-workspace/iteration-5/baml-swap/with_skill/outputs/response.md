**What I'm looking at.** The inference seam splits into four pieces: the port (the `Inference` tag and interface, already living inside `runtime-core`, shaped adapter-agnostically), the current live adapter (`inference.layer.ts` plus `baml_src/` plus the generated `baml_client/`, still sitting inside `runtime-core` rather than its own package), the planned extraction (`extract-inference-adapter`: proposal, design, and tasks all written, every task box unchecked, the target package holding only a placeholder export), and the hypothetical swap (`own-inference` — one clause in the roadmap ledger, no proposal file, no design doc, no stated Why anywhere in the repo).

Three constraints bound what either move can do. `extract-inference-adapter`'s own design doc already rules out the shortcut a swap-now plan would reach for: keeping `WelcomeResult` in the adapter's generated tree while the core imports it "recreates exactly the dependency this change exists to cut." A second open change, `framework-native-contracts`, plans to replace the string-named inference references the contract package uses today with direct typed function references, the exact shape an Effect-Schema adapter would need to target. It hasn't landed yet, either. A third constraint: `docs/toolchain.md`, dated three days before the extraction proposal and a week before the roadmap ladder that names `own-inference`, still names "BAML is never written by hand" as governing principle and describes a future where INDRA compiles permanently into generated BAML — a vision the later roadmap entry contradicts, with nobody having gone back to reconcile that page.

**What I know, and what would help to know.** Checked against the tree: the extraction proposal carries zero completed tasks, and the working copy stays clean, so nothing here has started yet. `own-inference` never got the proposal every sibling change earned. Every other entry under `openspec/changes/` opens with a substantive Why grounded in a concrete problem. This one has none. That absence carries its own signal: no commit, no note, no doc anywhere records an operational complaint against BAML — no cost figure, no latency number, no vendor risk, no build failure. What I can't verify from the tree: whether real pain drives the urge to swap, or whether the appeal runs purely architectural. Nobody can settle that, so I'll carry it forward as a stated assumption: no operational pain sits on record today, which reads the swap idea as a preference rather than a response to an incident.

**Which unknowns would actually move the recommendation.** Two candidates survive that filter. First: does real, currently unstated operational pain against BAML exist, strong enough to justify jumping the initiative's own stated order? Second: does `framework-native-contracts` land before anyone starts an Effect-Schema adapter, or does that adapter get built against the reference shape the change is about to retire? Both change what "commit to the swap now" costs, and they vary independently — urgency doesn't decide which change lands first, and landing order doesn't tell you whether the urgency exists. Two axes, so the full grid earns a walk.

**The four tiles, triaged.** Real pain paired with `framework-native-contracts` landing first reads as the cleanest tile: something concrete pushes the timeline, the team fast-tracks the contract reshape (already scoped, just unlanded), then points the Effect-Schema adapter at a stable target instead of a moving one. Real pain paired with racing ahead of `framework-native-contracts` still holds together as a story, but it spends the urgency on the wrong problem: the new adapter gets written against a reference shape about to change underneath it, so the pain that justified moving fast also guarantees a second rewrite. No pain paired with `framework-native-contracts` landing first collapses into "stay the current course" — finish the extraction, let the contract reshape land on its own schedule, and reopen `own-inference` later with a real Why. No pain paired with racing ahead earns explicit naming as the tile to avoid: it pays the double-rewrite cost from the second tile without even urgency to justify paying it.

**What holds no matter which tile arrives.** In every one of the four, the first concrete step reads the same: `WelcomeResult` has to leave the generated `baml_client` tree and land in `@indra/runtime-contracts` before any new adapter — BAML-based or Effect-Schema-based — can call itself done. Skipping `extract-inference-adapter` still performs that step, just unplanned, unreviewed, and without the four verification gates the extraction's own migration plan already wrote (core carries no BAML dependency, the adapter package becomes the sole BAML dependant, the offline suite plus the gated live test both stay green). "Commit to the swap now" and "finish the extraction first" read less like a fork than the question first posed, because the swap's own first move already lives inside the extraction plan.

**The hinges.** Two watchpoints would flip which tile is arriving, and both stay checkable without waiting for the swap itself. First: a proposal file appears at `openspec/changes/own-inference/` carrying a Why section that names a concrete BAML cost, latency figure, vendor risk, or build failure. That would confirm real pain over the assumption of none I'm currently carrying. Second: `framework-native-contracts` moves into `openspec/changes/archive/` before any `own-inference` directory exists — that would confirm the contract reshape lands first rather than the adapter work outrunning it.

**The call.** Finish `extract-inference-adapter` before writing a line of the Effect-Schema adapter, and specifically don't let the swap serve as the vehicle that performs the extraction's forced move informally. That holds up whether or not real pain shows up later, because the isolation work costs the same either way and the extraction's migration plan already carries the verification gates a rushed version would skip. This recommendation runs exposed in exactly one place: real, severe operational pain against BAML that outweighs the calendar cost of finishing the extraction properly turns waiting into a real cost. Watch the first hinge above for exactly that signal.
