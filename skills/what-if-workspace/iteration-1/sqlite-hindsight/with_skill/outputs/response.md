**Q1 — what am I actually looking at?** An internal admin tool, capped at around fifteen users, doing mostly reads: your team pulling up order statuses. That's the whole shape. No customers touch it, no SLA hangs off it, and nothing about it sounds irreversible either direction — this is about as low-stakes a database call as software makes.

**Q2 — what do I know, and what would I have to ask you to know the rest?** A few things settle without needing you at all. SQLite's own maintainers put the ceiling at "fewer than 100K hits/day should work fine... a conservative estimate, not a hard upper bound — demonstrated to work with 10x that" (<https://www.sqlite.org/whentouse.html>). Fifteen people checking order statuses isn't going to get near that in a hundred years. Writes aren't the bottleneck either: SQLite allows unlimited simultaneous readers and queues writers one at a time, and per that same page, "each application does its database work quickly and moves on, and no lock lasts for more than a few dozen milliseconds" — fine unless you have many writers hammering it at once, which fifteen people glancing at order statuses are not.

So the "would it have handled the load" question is settled, and it's settled in favor of over-engineered. But two things I can't verify from here change whether that verdict actually means anything for you, so I'll ask them and tell you the assumption I'm proceeding on for each:

- Does anything besides this tool's own UI talk to that Postgres database directly — a BI dashboard, another internal service, a data export job? If yes, "just SQLite" was never really on the table, because you'd be serving more than one consumer, and SQLite's own guidance is blunt about that: "avoid using SQLite in situations where the same database will be accessed directly... simultaneously from many computers over a network." *Assuming the answer is no* — this database exists for this one tool and nothing else queries it directly.
- Is the tool deployed as more than one running instance — multiple containers behind a load balancer, serverless functions, anything without one shared persistent disk? If so, over-engineered is the wrong word entirely: SQLite is a single file that wants one writer on one machine, and running it under a fleet of stateless replicas is a bigger operational headache than Postgres ever was, independent of your fifteen users. *Assuming the answer is no* — this runs as one instance, or at least one instance per environment, on a machine with a normal persistent disk.

**Q3 — is there anything left that would actually change the answer?** Given those assumptions, yes, one thing: whether this tool's shape changes going forward. Not "were you wrong in the past" — that's already answered — but "does the Postgres choice keep paying for itself." That's the one axis worth carrying, because it's the only unresolved thing that would flip what you do next.

Given one axis, this stays a light two-way split rather than a full scenario grid — the question doesn't earn more machinery than that, and "just curious lol" wasn't asking for a slide deck anyway.

**The two tiles.** In *Stays small*, the tool keeps doing exactly what it does today: your team, read-mostly, one instance, indefinitely. Here, yes — you over-built it. SQLite would've meant no managed instance to patch or pay for, no connection pool to configure, backups that are a file copy instead of a pg_dump job, and one less service that can page someone at 2am. None of that is a disaster today; Postgres for fifteen people just means you're carrying weight you didn't need to.

In *Outgrows it*, something changes the shape: the tool starts serving other teams or an external audience, it needs to run as more than one instance, or admins start hitting real write contention (bulk status updates landing concurrently, not just one person clicking a dropdown). Here the verdict flips — Postgres wasn't over-engineering, it was buying you a migration you never had to schedule. Ripping it out for SQLite now would've meant doing this exact same "should we move to Postgres" conversation later, under worse conditions, with real data and real users already depending on the tool.

**What holds regardless of which tile you're in:** the operational tax you're paying today — a managed instance, migrations, connection handling, auth — is real cost whether or not it was the right call in hindsight. That's worth naming even if you never touch this tool again: a managed Postgres instance for an internal fifteen-person tool is either dead weight (Stays small) or an early investment (Outgrows it), and only you know which bill you're currently paying.

**The hinge:** watch for either of two things — this tool being asked to serve a second consumer or team, or needing to run as more than one instance for deployment reasons. Either one lands you in Outgrows it and ends the "did we over-engineer" question in Postgres's favor. Neither happening in the next six months to a year is your signal that you're solidly in Stays small.

**Recommendation:** leave this specific tool alone. It's already built, the migration cost of tearing out a working Postgres setup outweighs the shelf space you'd save, and if you land in Outgrows it later you'll be glad you didn't just undo it. But bank the lesson forward instead of backward: the next time you or your team spins up an internal, single-team, single-instance tool like this one — no other consumers on the database, no plan to run it as multiple replicas — default to SQLite first and make Postgres the thing you reach for only when one of those two conditions shows up. That's the version of "did we over-engineer" that's actually worth answering, and it's a yes.
